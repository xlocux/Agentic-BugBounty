3 approcci architetturali

Quale struttura porta meglio a una UI che guida il workflow con controllo granulare?

A

Event Protocol — canale strutturato agente → UI

Il pipeline emette eventi JSON su stdout oltre ai log. La UI legge questi eventi (non il testo) per guidare il workflow.

{"type":"plan","phase":"researcher","operations":[
  {"id":"auth","label":"AUTH domain","mandatory":true},
  {"id":"inject","label":"INJECT domain","mandatory":true},
  {"id":"media","label":"MEDIA domain","mandatory":false,"skip_reason":"no file upload in scope"}
]}
✓ UI pausa, mostra il piano, aspetta approvazione → pipeline riprende
✓ Contratto formale: agente promette di emettere plan events prima di ogni fase
✓ Nessuna riscrittura del pipeline — solo aggiunge emit() in punti chiave
⚠ Richiede modificare run-pipeline.js per emettere eventi strutturati
⚠ IPC bidirezionale: UI deve anche inviare approvazioni al pipeline
B

Pipeline as API — il server orchestra le fasi direttamente

Ogni fase diventa una funzione Node.js chiamabile via API. Il server guida il workflow, non il pipeline monolite.

POST /api/phase/researcher/plan
→ {operations: [...], mandatory: [...]}

POST /api/phase/researcher/run
body: {approved_ops: ["auth","inject"]}
→ SSE stream con eventi strutturati
✓ UI è il vero orchestratore — controllo totale sul flusso
✓ Ogni fase testabile indipendentemente via API
⚠ Refactoring significativo: estrarre fasi da run-pipeline.js in moduli
⚠ Agenti Claude Code devono ancora girare come subprocess
⚠ Maggiore lavoro iniziale — 2-3 settimane di refactoring
C

Session File + Checkpoint Protocol ★ raccomandato

Il pipeline scrive un "session state" JSON strutturato a ogni checkpoint. La UI lo legge, mostra il piano, scrive l'approvazione. Il pipeline riprende.

// pipeline scrive:
session.json → {status:"awaiting_approval",
  phase:"researcher", plan:[...],
  selected_assets:[...]}

// UI scrive:
session.json → {status:"approved",
  approved_ops:["auth","inject"],
  assets:["*.duckduckgo.com"]}
✓ Minimo refactoring: il checkpoint system esiste già
✓ UI polling su session.json → notifica quando c'è un piano da approvare
✓ Bidirezionale: pipeline pausa, aspetta file, riprende
✓ Scala a opzione B in futuro senza buttare il lavoro
⚠ Latenza di polling (500ms max) — accettabile per workflow manuale
⚠ Serve schema formale per session.json
Raccomandazione: Opzione C ora → scala verso B in futuro. Il session file protocol riutilizza il checkpoint system esistente, aggiunge struttura senza riscrivere tutto. La UI diventa il centro di controllo senza toccare gli agenti CLI.
